📚 怎麼寫 Skill — Claude Code Skill 設計教學

2026-04-08 other source: ace + pa
TL;DR

Skill 是 Claude Code 的「角色 prompt 模組」——一份 SKILL.md 加上可選的 ref/ 與 memory/。寫 Skill 不是寫 code，是把可重用的工作流、判斷規則、輸出風格封裝成可被 /skill-name 觸發的指令。本指南涵蓋類型分類、結構規範、模型選擇、能力邊界、註冊流程與常見 pitfall。

① Skill 是什麼？跟一般 Prompt 差在哪

Skill 是 Claude Code 提供的「可命名、可觸發、可組合」的 prompt 模組系統，存放在 .claude/skills/<name>/SKILL.md。當你輸入 /<name> 或 Claude 自動匹配時，整份 SKILL.md 會被注入當前對話 context。

跟一般 prompt 的差別

  • 有命名空間：每個 Skill 有獨立目錄，可放 ref 文件、memory 索引、外部腳本，不會互相污染
  • 有觸發機制：使用者輸入 /skill-name 立即啟用；或在 description 中描述觸發條件讓 Claude 自動匹配
  • 有能力邊界：透過 [model: opus] 指定模型；透過角色設定限制工具用途（策劃型 vs 實作型）
  • 可組合：ACE 等調度型 Skill 可呼叫其他 Skill，形成 pipeline
  • 非強制：Skill 是「給模型參考」的軟約束。要強制執行請用 hooks（settings.json 的 PreToolUse/PostToolUse）
💡 關鍵分野
Skill = advisory（建議），模型可能忽略；Hook = deterministic（強制），harness 在 syscall 層攔截。需要「百分百執行」的規則寫 hook，不是 Skill。

② Skill 的四種類型

由 /pa（Prompt Architect）skill 定義的分類，依「自主性」與「部署形態」切分：

類型 A
純角色型 Skill
薄型跳板 → 一份 prompt 文件。沒有外部腳本、沒有自主決策，純粹切換對話風格與工作流。
範例：translator（中英翻譯）、weekly-report（週報模板）、coach（教練人設）
類型 B
工具型 Skill
CLI-First，封裝確定性邏輯。SKILL.md 主要說明可用指令，實作在 CLI binary 或 script。
範例：cloudflare（DNS 管理）、page（HTML 頁面選擇器）、todo（待辦速記）
類型 C
Agentic Skill
進階自主角色。多步驟決策、工具策略、子代理編排、Phase 結構、用戶確認檢查點。
範例：ace（任務調度）、pa（角色設計）、hawkeye（測試/QA）、scout（GitHub 偵察）
類型 D
Agent SDK Agent
獨立 TypeScript 應用程式，脫離 Claude Code 運行。用 @anthropic-ai/claude-agent-sdk 建立。
範例：排程任務、Webhook 回應、CI/CD 整合、長駐後台 Agent
💡 怎麼判斷自己要寫哪種
Q1：需要自己判斷下一步嗎？→ 否 = A 或 B；是 = C 或 D
Q2：涉及 CLI / 腳本邏輯嗎？→ 是 = B；否 = A
Q3：需要脫離 Claude Code 獨立部署嗎？→ 是 = D；否 = C
90% 的情況都是 A 或 C。新手從 A 開始。

③ SKILL.md 結構

最小可行 Skill（類型 A）

---
name: my-skill
description: 一句話描述這個 Skill 的能力與使用情境
---

[model: opus]

## 角色定位

你是 XXX，負責 YYY。

## 工作流程

1. 收到任務先做 ZZZ
2. 然後 ...
3. 最後產出 ...

## 溝通風格

簡潔、條列、不囉嗦。

## 任務結束

[初始設定 / 收尾動作]

ARGUMENTS: $ARGUMENTS

frontmatter 三個關鍵欄位

欄位必填說明
name✅Skill 名稱，對應 /skill-name 觸發指令。kebab-case，全小寫
description✅一句話描述。Claude 會用這段判斷該不該自動匹配。寫具體一點，含關鍵字
[model: xxx]建議body 第一行宣告。opus（複雜推理）/ sonnet（一般實作）/ haiku（簡單檢查）

body 常見區塊

  • 角色定位：你是誰、負責什麼、不負責什麼
  • 能力邊界：策劃型 Skill（spark/pm/uiux）禁止 Edit/Write；實作型才能改 code
  • 工作流程：Phase 0 → 1 → 2 → ... 結構，每 Phase 有明確輸入/輸出
  • 必經閘門：哪些情境必經哪些步驟（UI→必走 mockup、Code→必走 hawkeye）
  • 溝通風格：條列 vs 段落、問句頻率、表格用法
  • 初始設定：啟用後第一動（自我介紹、讀取記憶、列功能清單）
  • 任務結束：收尾規範（產出摘要、寫記憶、註冊報告）
  • ARGUMENTS：最後一行 ARGUMENTS: $ARGUMENTS 接收使用者帶的參數

④ 目錄結構與 lazy-loading

非平凡 Skill 的標準目錄：

.claude/skills/my-skill/
├── SKILL.md           ← 主 prompt（核心工作流）
├── MEMORY.md          ← 經驗索引（自動載入）
├── memory/            ← 個別記憶檔（按需 Read）
│   ├── feedback_xxx.md
│   ├── pattern_yyy.md
│   └── pitfall_zzz.md
└── ref/               ← 參考資料（按需 Read，省 token）
    ├── pipelines.md
    ├── skill-models.md
    └── matrix.md

什麼東西該放 ref/ 而不是 SKILL.md

  • 低頻參考表：規則表、矩陣、JSON 範例、配色清單——只在真正需要時 Read
  • 長條列：20+ 條的清單，主檔放摘要 + 連結到 ref
  • 多版本/分支邏輯：Pipeline 定義、Phase 變體
  • 原則：SKILL.md 主檔保持 <200 行；放 ref 的資料用 📎 詳見 `ref/xxx.md` 引用
💡 為什麼要 lazy-load
主 SKILL.md 每次啟用都會塞進 context，吃 token；ref/ 只在 Claude 判斷需要時才 Read。把不常用的細節抽到 ref 可大幅降低每次啟用成本。

⑤ 模型選擇 — 怎麼挑 opus / sonnet / haiku

模型適合場景對應 Skill 範例成本
opus複雜推理、多步驟決策、創意產出、調度編排ace, pa, spark, pm, uiux, scout, code-review高
sonnet程式碼編輯、結構化文件、一般實作frontend, backend, hawkeye, meeting-minutes, weekly-report中
haiku檔案掃描、MCP 轉發、簡單分類、批次處理doc-checker, todo, page低

調度時的成本優化

  • opus Skill 用 Skill tool（共享主對話 context，無額外 sub-agent 成本）
  • sonnet/haiku Skill 用 Agent tool（指定 model 參數，省成本）
  • 例外：需要大量 context 的 sonnet Skill（如 hawkeye 看完整 codebase）可用 Skill tool 但要標註理由

⑥ 能力邊界 — 策劃型 vs 實作型

這是 Claude Code 慣例的硬規則：策劃型 Skill 不准呼叫 Edit / Write 修改原始碼。

類別能力對應 Skill
策劃型產出方案/文件/設計，禁止 Edit/Write 改 codespark, pm, uiux, game-master, data-architect
實作型可呼叫 Edit/Write 改原始碼frontend, backend, godot, html-viewer
調度型不直接產出，只調度其他 Skillace
分析型產出報告、摘要、研究，可用 Read/Grep/Webscout, hawkeye, dual-brain, code-review

寫 Skill 時必須在角色定位明確標示自己屬於哪類，並寫上「⛔ 不得 ...」明確邊界，避免模型越權。

⑦ Skill vs Hook — 什麼該寫哪個

需求用 Skill用 Hook
給 Claude 一個工作風格 / 角色 / 流程模板✅❌
百分百攔截某類工具呼叫（讀 secret、刪檔）❌（會被忽略）✅（PreToolUse）
每次 commit 前自動跑 lint/test❌✅（PreToolUse Bash matcher）
偵測指令模式並記錄統計❌✅（PostToolUse）
規範產出格式（HTML 一律淡色主題）✅（軟提醒）⚠ 可加 PostToolUse 補強
使用者輸入時自動掃描敏感字串❌✅（UserPromptSubmit）

原則：能 Skill 解決的優先 Skill（無侵入、易維護）；Skill 解不掉再加 hook。Hook 是程式碼層的硬規則，跑不過會直接 exit 2 阻擋。

⑧ 從零寫一個 Skill 的步驟

  1. 啟動 PA 引導 輸入 /pa 開始需求訪談。PA 會問你角色名稱、定位、使用場景，並判斷類型（A/B/C/D）。
  2. 需求澄清 回答 PA 的問題：核心能力、預期輸入輸出、邊界條件、有沒有參考的現有 Skill。新手選類型 A 即可。
  3. Step 1 草稿 PA 用三步驟工作法產出第一版 prompt。包含 [角色]、[任務]、[功能]、[工作流程]、[初始設定] 等區塊。
  4. Step 2 批判 PA 自己審查草稿：結構完整性、邏輯一致性、安全性、可執行性、格式規範等 9 個維度逐項打分。
  5. Step 3 重寫 針對批判結果重寫。最多迭代 3 輪。
  6. Step 4 資源審查 檢查 prompt 體積、模型配置、lazy-loading 機會、成本控制。>3000 tokens 須評估瘦身。
  7. 交付與註冊 儲存 SKILL.md、更新 CLAUDE.md 角色清單、若是新模組同步更新 dashboard 與 MEMORY.md。
  8. 實測迭代 實際跑幾次任務，觀察 Skill 的行為。發現問題回頭用 /pa /優化 my-skill 直接優化。

⑨ 註冊與連動更新

新 Skill 不只是建檔，還必須連動更新所有相關位置：

位置動作
.claude/skills/my-skill/SKILL.md建立主檔
CLAUDE.md 角色清單加一行（名稱 + 一句話描述）
memory/MEMORY.md若有專用記憶，加索引指標
dashboard skills 統計新 layer 要更新 SKILL_LAYERS 對應
對應的 hook（如 skill-usage-tracker）確認新 Skill 名稱會被自動追蹤
💡 PA 會自動處理
用 /pa 走完交付流程的話，PA 會自動更新這些連動位置。手動寫 Skill 才需要自己一個一個改。

⑩ 常見 Pitfall

⚠️ Pitfall 1 — Skill 過度膨脹
把所有規則塞進 SKILL.md，越長越詳細。結果每次啟用都吃 3000+ tokens，主 context 容易爆。
解法：主檔 <200 行；低頻細節抽 ref/，用 📎 詳見 引用。
⚠️ Pitfall 2 — 邊界模糊
沒寫「不得做什麼」，模型自由發揮越權做事（策劃型 Skill 跑去改 code）。
解法：角色定位明寫 ⛔ 你不寫 code、不做設計、不寫測試，列負面清單。
⚠️ Pitfall 3 — 跟現有 Skill 重疊
沒檢查就建新 Skill，後來發現 ace/pa/spark 已涵蓋類似功能，造成兩條規則打架。
解法：建之前先 grep 現有 SKILL.md，找關鍵字。能擴充現有的就不要新建。
⚠️ Pitfall 4 — Context Collapse
單一 Skill 同時承載 5+ 個角色 context，注意力稀釋導致行為不穩定。
解法：角色密集任務拆 sub-agent，主 Skill 只做調度；或用 Phase 結構切階段，每階段 context 聚焦。
⚠️ Pitfall 5 — 規則清單過長 (Logic Collapse)
同一推理步驟塞 10+ 約束條件，模型逐條檢查容易漏。
解法：拆成獨立驗證步驟，逐步收斂；或內化為「思維模式」（分類本能、反轉思考、偏執掃描）而非清單。
⚠️ Pitfall 6 — 把 Skill 當成強制執行層
寫「絕對不准 X」「必須一定要 Y」期待 100% 遵守。但 Skill 是 advisory，模型可能忽略。
解法：真正硬規則寫 hook（settings.json）；Skill 只負責「如何做得好」而非「強制不准」。

⑪ 進階：經驗沉澱與自我進化

MEMORY.md 的角色

每個 Skill 可有自己的 MEMORY.md 索引（簡短一行條目），啟用 Skill 時自動載入。完整記憶內容放在 memory/ 子目錄，按需 Read。這樣可以讓 Skill 跨 session 累積經驗而不污染 context。

記憶分類

  • pitfall — 使用者修正了你的決策
  • pattern — 確認了非顯然的有效做法
  • decision — 重要的設計取捨（為什麼選 A 不選 B）

任務結束自動沉澱

在 SKILL.md 的「任務結束」段落寫自動檢查機制：

## 任務結束

任務交付後，靜默檢查本次是否有值得記錄的經驗：
- pitfall — 使用者修正了 ...
- pattern — 確認了非顯然的 ...
- decision — 重要取捨 ...

有 → 寫入 memory/ 並更新 MEMORY.md
無 → 不提

用 /pa /優化 自我迭代

當你發現 Skill 行為不對，用 /pa /優化 my-skill 直接進入批判→重寫流程，PA 會讀現有 SKILL.md 並提出改進建議。
