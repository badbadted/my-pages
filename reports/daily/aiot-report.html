⏱ 30 分鐘

ESPHome 2026.4.x — 感測器發布速度 46 倍提升 + OTA 簽名安全
⏱ 1 小時
2026.4.0 的核心改進：感測器資料發布速度最高提升 46 倍（log 格式化從裝置端移至客戶端），ESP32 預設 240MHz（API 操作快 33%）。OTA 簽名驗證讓韌體更新更安全，不需要硬體 Secure Boot。GPIO 擴展器中斷驅動使 I2C 閒置流量減少 99.7%——直接影響 FP2/Zigbee 感測器的即時響應精度。今日（2026-05-21）爬取確認尚無 2026.5 版本，此版仍為最新穩定。
滾動保留 v2026.4.x（目前最新穩定，今日確認） ⚠️ Breaking：i2s_audio media player 移除；電池裝置需加 cpu_frequency: 160MHz 📋 Changelog →
操作步驟
  1. 電池供電裝置先確認：升級前在 YAML 加 esp32: framework: ... cpu_frequency: 160MHz，避免耗電增加
  2. ESPHome Dashboard → 每個裝置 → Update，逐一 OTA 更新（優先更新常時插電 ESP32 裝置）
  3. 有使用 i2s_audio media player 的設定需遷移至 speaker 元件（查 Breaking Changes 頁）

llama.cpp b9253 統一執行檔（今日最新）— 簡化 RTX 5070 Ti 本地 LLM 部署
⏱ 1–2 小時
你的已知痛點：「Local AI for intent recognition（reduce Gemini API dependency）」。今日（2026-05-21）最新 b9253 帶來重大易用性改進：統一執行檔架構（unified executable）——原本分散的 llama-cli / llama-server / llama-bench 整合成單一 llama 指令，llama server / llama run 一鍵啟動。這大幅降低在 Windows（192.168.0.229）上建立本地推論 API 的門檻。RTX 5070 Ti 可跑 Qwen3.5-14B-Q4 做中文意圖辨識，替代部分 Gemini function calling。上次 b9204 → 今日 b9254，3 天 +50 個版本，活躍度極高。

操作步驟
  1. 在 192.168.0.229 更新 llama.cpp：git pull && cmake -DGGML_CUDA=ON -B build && cmake --build build -j --config Release
  2. 使用新統一指令啟動推論服務：./build/bin/llama server -m Qwen3.5-14B.gguf -ngl 99 --port 8080
