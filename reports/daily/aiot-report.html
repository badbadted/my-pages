升級 Home Assistant Core 至 2026.5 — 維護儀表板 + RF 射頻支援（你落後兩版）
⏱ 1–2 小時
本期第一優先。你的 HA Container 停在 2026.3.1，而官方已發行 2026.5（5/6）。最實用的是 維護儀表板（Maintenance Dashboard）：一畫面列出所有低電量裝置 —— 對你 178 個 FP2 實體 + Zigbee 電池裝置群的維運直接有感。另有 RF 射頻裝置支援（捲門/車庫門可入 HA）與 ESPHome 序列埠網路化。
問題 / 機會
落後兩版意味著錯過維護儀表板、12 項新整合與多項自動化編輯器改善；版本差距越大，未來一次跳升的整合破壞風險越高。
建議做法
分兩跳升級 2026.3.1 → 2026.4（IR 一級公民）→ 2026.5，每跳後驗證整合。
🆕 本期新增 2026.5（5/6 發行） ⚠️ 跨兩版，先備份再升 📋 HA Blog →
操作步驟
  1. 備份：HA → Settings → System → Backups → 立即建立完整備份
  2. 先升 2026.4：docker pull homeassistant/home-assistant:2026.4 && docker restart homeassistant，驗證 FP2 / Z2M / ESPHome / WLED 整合無紅字
  3. 再升 2026.5：改 tag 為 :stable 重啟，開啟維護儀表板確認低電量裝置清單正常
  4. 逐一檢查 4 個核心整合的實體狀態與既有自動化（迎賓 TTS / 自動關冷氣）仍運作
🟡 建議先測試 · 下次維護視窗執行 相關服務：192.168.0.222 HA Container (Docker/OrbStack)
ESPHome 升級至 2026.5.0 — 省電模式電流近半 + Device Builder 新介面
⏱ 1–2 小時
主迴圈與排程器架構重大改進，省電模式電流從 2.0 mA 降至 1.1 mA（電池供電 ESP 裝置續航直接受益）；修正 watchdog feed 率 1000× 回歸（CPU 開銷 46.5ms → 21.6ms）。另推全新 Device Builder Web 介面（視覺化元件 + 韌體排程佇列）取代舊 Dashboard，並新增 ESP32-C6 / H2 的 Zigbee 感測器曝露。
問題 / 機會
你的 ESPHome 裝置群升級後待機功耗與 CPU 開銷明顯下降；C6/H2 Zigbee 支援讓未來 ESP32 直接走 Zigbee 不必額外橋接。
🆕 本期新增 2026.5.0 ⚠️ modbus_controller server 模式 → 改 modbus_server 📋 ESPHome Changelog →
操作步驟
  1. HA Add-on（ESPHome Device Builder）更新至 2026.5.0
  2. 若有用 modbus_controller server 模式，先改成新的 modbus_server 元件再重編
  3. 逐台 OTA 重編韌體；電池裝置確認省電模式生效後再批次推
🟡 建議先測試 · 單台驗證後批次推 相關服務：192.168.0.222 HA + ESPHome 裝置群
Zigbee2MQTT 升級至 2.10.1 — Tuya 屬性熱修復 + HA 群組實體進 discovery
⏱ 30 分鐘
2.10.1（5/7）是熱修復版，解決 多個 Tuya 裝置屬性無法更新 / 寫入 的問題（影響所有 Tuya Zigbee 用戶）；2.10.0 起 HA 群組實體會自動寫入 discovery 配置，並擴充 Aqara W600（電池 / 閥門告警 / 排程）、Shelly Gen4、Yale 門鎖。共 18 款新裝置。
問題 / 機會
若你有 Tuya Zigbee 裝置，2.10.1 直接修掉狀態不同步；HA 群組 discovery 省去手動 YAML。
🆕 本期新增 2.10.1（5/7） 官方 changelog 頁持續 404，以 GitHub Releases 為準 📋 GitHub Releases →
操作步驟
  1. 記下幾個 Tuya 裝置目前狀態值供升級後對比
  2. HA Add-on Store 直接升級，或 docker pull koenkk/zigbee2mqtt:latest && docker restart z2m
  3. 觸發裝置開/關/亮度，確認 logs 無屬性寫入錯誤；HA 群組實體正常出現
🟢 低風險 · 下次維護視窗執行 相關服務：192.168.0.222 Zigbee2MQTT
Frigate 升級至 0.17.1 — 安全修復：限制 raw config 端點 + 修正跨攝影機鑑權
⏱ 1 小時
0.17.1 為安全維護版：raw config 端點限定 admin 角色、修正 timeline / media 端點的跨攝影機鑑權漏洞、exec/expr/echo go2rtc 來源預設停用。你的 Frigate NVR 對外暴露時這兩項鑑權修復值得儘快套用。
問題 / 機會
跨攝影機鑑權漏洞可能讓低權限使用者越權存取其他攝影機畫面；屬可儘快處理的安全項。
🆕 本期新增 安全相關 ⚠️ 若從 0.16 升，genai 設定需遷移至 objects → genai 📋 GitHub Releases →
操作步驟
  1. 備份 Frigate config；若仍在 0.16，先把 genai 區塊遷移到 objects → genai
  2. 升級容器並重啟，確認攝影機串流與物件偵測正常
  3. 驗證低權限帳號無法存取 raw config 與其他攝影機端點
🟡 安全項 · 建議近期執行 相關服務：Frigate NVR + RTX 5070 Ti（GenAI 加速）
評估 OpenJarvis 本地個人 AI 框架 — 降低 Luna 對 Gemini API 的依賴
⏱ 半天評估
Ollama 5/28 發布 OpenJarvis v1.0：開源框架，建構跑在自有硬體上的個人 AI 代理，內建 Ollama 支援、全程本地執行。契合你「本地意圖辨識、減少 Gemini API 依賴」的痛點 —— 可用 RTX 5070 Ti（192.168.0.229）跑簡單指令路由（開燈/關燈/查狀態），複雜推理才上雲。
問題 / 機會
Luna 18 個 function call 全走 Gemini，簡單指令也付雲端費用；本地框架可吃掉高頻低複雜度的請求。
🆕 本期新增 OpenJarvis v1.0（5/28） 📋 Ollama Blog →
評估步驟
  1. 在 192.168.0.229 跑 Ollama + qwen3，部署 OpenJarvis 試其代理架構
  2. 挑 3–5 個 Luna 高頻簡單指令，比對本地 vs Gemini 的準確率與延遲
  3. 若達標，在 Luna server.js 加本地意圖路由分流；否則記錄結論待模型升級再評
⚪ 評估觀察（潛在 API 費用節省） 相關服務：192.168.0.229 RTX 5070 Ti + Luna server.js

⏱ 1–2 小時
b9415（5/29）帶來 DeepSeek V3.2 稀疏注意力（lightning indexer） 改善長上下文記憶體與吞吐、DeepSeekOCR 2 多模態支援、flash attention 改用 f16 mask 降記憶體、server timeout 拉到 3600 秒。續用 CUDA 13.1 Windows 建置對應 Blackwell。是 Card 5 本地推論路線的底層引擎。
